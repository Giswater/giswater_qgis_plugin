This file explains the SQL structure of dbmodel/.
For full architecture, manifests, CLI, and testing see README.md.


-- Folders structure
--------------------------------
manifests          YAML build pipelines (ws, ud, utils, am, cm, audit, cibs)
schemas/
  main/            Network project schemas (ws, ud)
    common/        Shared SQL loaded into BOTH ws and ud (not a PG schema)
    ws/
    ud/
  addon/           Satellite schemas
    utils/
    am/
    cm/
    audit/
    cibs/
corporate/         Optional corporate/custom overlays (outside manifests)
dev/
docker/
test/              pgTAP tests + Docker harness
tools/

Builds are driven by manifests/ + giswater_admin (same engine as QGIS Manage Schemas).
See also: ../giswater_admin/README.md


-- Subfolder structure (schemas/main/ws and schemas/main/ud)
--------------------------------
base/
  init.sql         (common only)
  fct/             Functions (one file per function)
  ftrg/            Trigger functions (one file per trigger function)
  schema_model/    Base DDL, views, rules, table constraints, triggers
updates/
  M/m/p/
    patch.sql      All SQL for that version bump (single file per scope)
    changelog.txt
sample/            Optional seed data (user/, inv/, dev/)
final_pass/
  config_form_fields/
  i18n/            Locale folders (en_US, es_ES, ca_ES, …)


-- Subfolder structure (schemas/main/common)
--------------------------------
base/
  init.sql
  fct/
  ftrg/
updates/
  M/m/p/
    patch.sql
    changelog.txt


-- Subfolder structure (schemas/addon/<kind>)
--------------------------------
base/
  init.sql
  base.sql         (DDL bootstrap where applicable)
  fct/
  ftrg/
integration/       Parent-link SQL (utils, cm, audit, cibs)
  common/          (cm, audit)
  ws/ | ud/        Type-specific integration hooks
updates/
  M/m/p/
    patch.sql
    changelog.txt
sample/            (optional, e.g. cm)
i18n/              (optional, e.g. cm, am)


-- Update order (network ws/ud)
--------------------------------
For each version M.m.p the engine applies, in order:
  1) schemas/main/common/updates/M/m/p/patch.sql
  2) schemas/main/ws/updates/M/m/p/patch.sql   (or ud/updates/…)
Then moves to the next version.

Satellite schemas walk a single updates/ tree:
  schemas/addon/<kind>/updates/M/m/p/patch.sql


--- Instructions to update SQL
--------------------------------
1) FUNCTION AND TRIGGER FUNCTION (base):
   Edit the original definition in base/fct/ or base/ftrg/.
   They must be unique; use one file per function/trigger.
   In case of different code for different versions use 'IF' in code.
   Do NOT duplicate base definitions in update patches unless strictly needed.

2) VIEWS:
   Do not modify schema_model/ view definitions for version bumps.
   Put view changes in the version patch.sql for the correct scope (common, ws, ud, …).
   DROP is allowed when needed; DROP CASCADE is FORBIDDEN (wait for next major if unavoidable).
   Document any view change in changelog.txt.

3) TABLES, RULES, TRIGGERS, CONSTRAINTS:
   Do not modify schema_model/ base definitions for version bumps.
   Put changes in patch.sql under the correct updates/M/m/p/ folder.
   DROP is FORBIDDEN. If a relation becomes deprecated use:
     UPDATE audit_cat_table / audit_cat_function / audit_cat_sequence SET isdeprectaded=TRUE

4) DML:
   Same as above: patch.sql in the correct scope folder.
   For inserts on system tables prefer ON CONFLICT (pk_field) DO NOTHING.

5) I18N:
   Network (ws/ud): schemas/main/{ws,ud}/final_pass/i18n/<locale>/
   CM: schemas/addon/cm/final_pass/i18n/<locale>/
   AM: schemas/addon/am/final_pass/i18n/<locale>/
   Do not edit en_US base strings in place for version bumps; add/update via patch.sql or
   locale files as appropriate for the manifest final_pass / load_locale phase.

6) NEW VERSION BUMP (network):
   Create folders as needed:
     schemas/main/common/updates/M/m/p/patch.sql + changelog.txt   (if shared)
     schemas/main/ws/updates/M/m/p/patch.sql + changelog.txt         (ws-only)
     schemas/main/ud/updates/M/m/p/patch.sql + changelog.txt         (ud-only)
   Use a single patch.sql per scope (no separate ddl.sql, dml.sql, ddlview.sql, trg.sql for new bumps).

7) CORPORATE / CUSTOM:
   SQL under corporate/ is outside standard manifests.
   Keep corporate overlays separate from schemas/main and schemas/addon.


-- Changelog file
-----------------------------
  - Use it to register any change on SQL.
  - Issue is mandatory. If it does not exist, create a new one on GitHub.
  - Network (ws/ud): one changelog.txt per scope under schemas/main/{common,ws,ud}/updates/M/m/p/
      * common: changes shared by ws and ud
      * ws or ud: changes only for that project type (same version folder as patch.sql)
  - Satellite: changelog.txt beside patch.sql under schemas/addon/<kind>/updates/M/m/p/
  - Preferred format: one bullet per line, lines starting with "- "
      * Optional comments with "#"
      * Legacy format (version line + asterisks + bullets) is still accepted
  - Manage Schemas dialog and `giswater_admin update --check` merge common + ws|ud for display.
  - If a view must be dropped without cascade, document it explicitly in changelog (special case).


-- Rules to prevent conflicts on update files
--------------------------------------------
  - For ct and trg use DROP...IF EXISTS......CASCADE
  - For tables and sequences use CREATE TABLE/SEQUENCE IF NOT EXISTS ....
  - For views use CREATE OR REPLACE VIEW
  - For insert on system tables use ON CONFLICT (pk_field) DO NOTHING
  - For new fields, USE THE FUNCTION gw_fct_admin_manage_fields (example):
      SELECT gw_fct_admin_manage_fields($${"data":{"action":"ADD","table":"config_web_fields", "column":"table_type", "dataType":"text"}}$$)


-- Workflows (via manifests + giswater_admin / Manage Schemas)
----------------------------
1) CREATE EMPTY PROJECT          profile empty
2) CREATE PROJECT WITH SAMPLE    profile sample_full
3) CREATE PROJECT SAMPLE + INV   profile sample_inv
4) CREATE PROJECT FOR DEV        profile dev
5) PROJECT UPDATE                profile update
6) CREATE / INTEGRATE SATELLITE  utils, cm, audit, cibs (see manifests/*.yaml)
7) LAST PROCESS FUNCTION         gw_fct_admin_schema_lastprocess (manifest phase lastprocess)
   - Grant permissions to all relations using audit_cat_* tables
   - Enable foreign keys with utils schema if exists
   - Drop deprecated table/views/functions/sequences ONLY for new projects
